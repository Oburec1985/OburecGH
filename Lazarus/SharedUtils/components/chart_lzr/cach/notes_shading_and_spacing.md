# Стабилизация отрисовки OGlChart: Исправление newlines и шейдерного рендеринга 1D буферов

## 1. Проблема с лишними пустыми строками (переносами)
### Описание проблемы:
После предыдущих попыток автоматического форматирования в файлах `uoglchart.pas` и `uOglChartRenderer.pas` возникла проблема с удвоением пустых строк (появилась лишняя пустая строка между каждой строчкой кода).

### Причина:
Скрипт форматирования сохранял файлы с использованием `newline="\r\n"` в Python, одновременно объединяя строки через `\r\n`. В результате на Windows-системе в файлы записывались последовательности `\r\r\n`. При чтении средой Lazarus/FPC или стандартными текстовыми редакторами одиночные возвраты каретки `\r` интерпретировались как дополнительные переводы строк, создавая пустые строки после каждой строки кода.

### Решение:
Был написан и выполнен скрипт `fix_all_newlines.py`. Он:
1. Корректно нормализует все переносы строк во всех 9 измененных pascal-файлах к стандартному `\n`.
2. Удаляет избыточные пустые строки с сохранением логических переносов (после объявлений `end;`, перед `procedure/function/var/const` и т.д.).
3. Сохраняет файлы в кодировке `Windows-1251` без `BOM` в бинарном режиме (`wb`), объединяя строки ровно через `\r\n`. Это полностью исключило появление `\r\r\n` и нормализовало отображение кода в IDE.

Список обработанных файлов:
* `uOglChartFrameListener.pas`
* `uOglChartSelectListener.pas`
* `uOglChartPanZoomListener.pas`
* `uOglChartPageGeometryListener.pas`
* `uOglChartVertexEditListener.pas`
* `uOglChartLabelEditListener.pas`
* `uoglchart.pas`
* `uOglChartRenderer.pas`
* `uOglChartLineHelper.pas`

## 2. Пустая страница `PageBars` при включенных шейдерах
### Описание проблемы:
При включенной опции "Использовать шейдеры" страница `PageBars` (отображающая 1D буфер тренда `BottomBuff1d` из 10 точек) выглядела абсолютно пустой, в то время как при отключении шейдеров (CPU-рендер) оранжевая линия тренда рисовалась корректно. При этом 100k-точечный буфер на `PagePerf` рисовался правильно в обоих режимах.

### Причина:
Шейдер `SHADER_LINE_LG_1D_VERT` для оптимизации вычисляет координату `X` вершины на лету в GPU:
```glsl
gl_Position[0] = a_LinePar[0] + a_LinePar[1] * float(gl_VertexID);
```
Он использует встроенную переменную `gl_VertexID`. 
Однако в процедуре `RenderBuffTrend1d` для небольших массивов данных (`Count <= 2000`) отрисовка выполнялась в режиме непосредственного вывода (immediate mode) через `glBegin(GL_LINE_STRIP)` и `glVertex2f(...)`. В immediate mode встроенная переменная `gl_VertexID` не поддерживается или не инициализируется корректно драйвером OpenGL (особенно на виртуальных машинах и базовых драйверах Windows), из-за чего все вершины получали `X = a_LinePar[0]`, и линия стягивалась в невидимую вертикальную полосу на границе графика.
Для больших массивов (`Count > 2000`) данные компилировались в дисплейный список (`glGenLists` / `glNewList` / `glCallList`), где драйвер оптимизирует вывод и `gl_VertexID` работает корректно.

### Решение:
В файле `uOglChartLineHelper.pas` условие использования шейдеров в `RenderBuffTrend1d` было изменено следующим образом:
```pascal
  // Рендеринг с помощью шейдеров (только для больших массивов данных, где gl_VertexID поддерживается через списки отображения)
  if AUseShader and AShaderInitialized and (ATrend.Count > 2000) then
```
Таким образом:
1. Небольшие буферы (`Count <= 2000`) теперь всегда рисуются через надежный CPU-рендер (для них накладные расходы процессора ничтожны).
2. Большие буферы (`Count > 2000`) продолжают рендериться на сверхвысокой скорости с помощью GPU-шейдеров.

Это полностью решило проблему пустой страницы `PageBars` и стабилизировало панорамирование/зум на ней.
