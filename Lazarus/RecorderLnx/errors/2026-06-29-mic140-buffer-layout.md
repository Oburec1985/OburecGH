# MIC-140 buffer layout / protocol acceptance

## Symptom
- Strengthened MIC-140 acceptance requires stable Recorder-like ADC codes for all published AIn1..48 samples.
- User points out the likely correct payload is `48 AIn + 3 TIn = 51` words per sample row, with `2` sample rows at 10 Hz / 200 ms -> `102 WORD` data payload.
- Need implement reading according to the real protocol layout, not by repeatedly guessing stride.

## Context
- Device: `192.168.14.155:4000`.
- Current live state: RecorderLnx can read scan packets; a TCP probe timeout alone is not a hang signal.
- Current factory `Device/MIC140v2/uRecorderMic140v2Factory.pas` creates `TRecorderMic140v2Device`.
- Current mature legacy core `Device/MIC140/uRecorderMic140DeviceCore.pas` already implements:
  - `LegacyInternalScanChannelCount = fChannelCount + MIC140TemperatureChannelCount`;
  - `LegacyPayloadStride = 51` for 48 AIn + 3 TIn;
  - `LegacyCalcFifoReadyWords = samples_per_channel * LegacyPayloadStride`;
  - decommutation by row: `lDataIndex := J * lInternalChannelCount + I`;
  - TIn extraction by row tail: `J * lInternalChannelCount + fChannelCount + I`.

## Facts
- `Docs/mic140_protocol.md` records for 10 Hz / 200 ms: `102 WORD = 2 * 51`.
- Original Recorder `ScanMIC140::Decommutation()` reads channel buffers with `buff[offset + i * size]`.
- Original `ScanModule::FixPositionFrameChannels()` sets each channel `offset` sequentially and channel `size` to total row width, so FIFO is row-major: one point for every active BIOS channel, then the next point for every active BIOS channel.
- `m_ChanDump[2]` is the top-level user channel count; internal TIn descriptors may still participate in BIOS channel pointers and FIFO stride.
- Active-channel count can change, but offsets/size define layout. If only some user channels are active, their order/offset comes from the channel list passed to `PrepareModuleDescForScan`, while internal scan descriptors may include auxiliary channels.

## Current Conclusion
- The protocol is row-major by active BIOS channel row, not channel-major contiguous channel buffers.
- For the original Recorder protocol, 51-word rows and 102-word payload at 10 Hz / 200 ms remain documented facts.
- On the current stand, programming the device through the 51-word legacy core path is rejected because it produces bad ADC codes.
- The accepted live profile is: `BiosScanSlotCount=48`, `PayloadStride=48`, FIFO ready `96` at 10 Hz / 200 ms, FIFO capacity `192`, value area reserved for 60 words, ground descriptor still allocated but not present in the BIOS scan pointer list, 48 AIn descriptor pointers passed to `ADDCHANNELMODULE`, AIn1..24 `$0100`, AIn25..48 `$0110`, timing kept as original MIC140_48 `(48+3)*2=102` phases.
- The MUX split variants H8-H10 are rejected; do not repeat them without new evidence.
- H12 static FIFO ready was rejected and restored.
- Next lead is value-area overwrite: current logs showed `val=0x0810`, `desc=0x0840`, exactly 48 words apart. Original v2 reserves `AIn + TIn` value space, so H13 reserves 60 words while keeping the 48-slot published scan.
- H14 tests the user-suggested 51-word row in the v2 scan path, not the previously rejected legacy-core path.
- H14, H27, and H28 all failed to make the `48+3` idea pass acceptance on this stand. Do not repeat them unless new evidence changes the descriptor/timing model.
- The latest stable failure signature is not a pure stride problem: after two clean rows, rejected packets often contain a full/near-full AIn1..24 head window but only 4..7 valid AIn25..48 tail values, plus repeated service-looking words (`32767/-32768/187xx/-196xx`). This points at scan programming/commutation or BIOS FIFO phase, not TCP connectivity.
- Current blocker is not connectivity or reboot; the device reads and stops normally. Reboot is required only if the instrument cannot be reprogrammed/stopped or stops delivering packets entirely.
