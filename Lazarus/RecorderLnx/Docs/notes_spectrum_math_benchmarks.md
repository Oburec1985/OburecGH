## Условия теста

- Компилятор: Free Pascal 3.2.2, сборка через `C:\lazarus\lazbuild.exe`.
- Целевая платформа: Windows x64.
- Размер указателя: 64 bit.
- Оптимизация проекта: `-O3`.
- Выравнивание рабочих массивов: 64 байта.
- Тестовый сигнал: сумма нескольких гармоник с разными амплитудами и фазами.
- Окно: Hann.
- Проверяемые backend:
  - `DFT_Pascal` - прямое дискретное преобразование Фурье, используется как эталон корректности.
  - `FFT_Iterative_Pascal` - итеративное radix-2 FFT на Pascal, текущий чемпион и базовый эталон для сравнения новых backend.
  - `FFT_Iterative_SSE3_RMS` - тот же FFT, но расчет RMS-амплитуд выполнен через SSE3 inline assembler.
  - `FFT_Iterative_AVX_RMS` - тот же FFT, но расчет RMS-амплитуд выполнен через AVX/YMM inline assembler, по две комплексные точки за итерацию.
  - `FFT_Precomputed_Pascal` - FFT с заранее рассчитанными bit-reverse индексами и twiddle factors, RMS считается Pascal-кодом.
  - `FFT_Precomputed_AVX_RMS` - precomputed FFT плюс AVX/YMM расчет RMS-амплитуд.

Важно: `FFT_Iterative_SSE3_RMS` и `FFT_Iterative_AVX_RMS` ускоряют только пост-FFT расчет RMS-амплитуд. `FFT_Precomputed_Pascal` и `FFT_Precomputed_AVX_RMS` уже меняют само FFT-ядро: убирают `SinCos` из горячего цикла и используют заранее построенный план.

## FFT-only runtime: сравнение технологий

Этот замер соответствует runtime-режиму после инициализации:

- FFT-план уже построен;
- экспоненциальные множители уже рассчитаны и лежат в плане;
- RMS не считается;
- окно не входит в измеряемую FFT-функцию;
- измеряется только runtime FFT: bit-reverse загрузка входа и butterfly.

| FFT size | Pascal precomputed, us | Pascal inline, us | SSE2 scalar asm, us | AVX scalar asm, us | AVX vector2 asm, us | Winner |
|---:|---:|---:|---:|---:|---:|:---|
| 256 | 3.277 | 2.458 | 4.301 | 5.530 | 3.277 | Pascal_Inline_Precomputed |
| 512 | 7.578 | 5.939 | 9.216 | 8.806 | 7.168 | Pascal_Inline_Precomputed |
| 1024 | 15.565 | 13.107 | 19.456 | 19.456 | 14.131 | Pascal_Inline_Precomputed |
| 2048 | 34.816 | 27.648 | 44.032 | 43.008 | 28.672 | Pascal_Inline_Precomputed |
| 4096 | 77.824 | 60.416 | 95.232 | 92.160 | 78.848 | Pascal_Inline_Precomputed |
| 8192 | 232.107 | 146.773 | 208.213 | 208.213 | 143.360 | AVX_Vector2_InlineAsm_Precomputed |

Все варианты дали `DiffVsPascal=0`, то есть результат совпал с базовым `Pascal_Precomputed`.

Вывод по технологии: на малых и средних размерах победил `Pascal_Inline_Precomputed`. На 8192 точках победил полноценный AVX/YMM-кандидат `AVX_Vector2_InlineAsm_Precomputed`, который обрабатывает две complex-бабочки за раз (`143.360 us` против `146.773 us`). Скалярный SSE/AVX asm по-прежнему проигрывает.

AVX-512 проверен отдельно: установленный FPC 3.2.2 x86_64-win64 не распознает регистры `zmm0` в inline assembler, поэтому AVX-512 backend через текущий Lazarus/FPC inline asm недоступен. Для AVX-512 нужен внешний объектный модуль/C ABI или другой ассемблерный путь.

## Batch FFT runtime: до 128 спектров

Этот замер соответствует практическому режиму, когда на одном ПК нужно считать много независимых спектров. Используется текущий быстрый runtime FFT `Pascal_Inline_Precomputed`. План построен заранее, RMS не считается. На тестовом ПК обнаружено `16` логических CPU.

Сравниваются режимы:

- `Single` - последовательный расчет всех спектров одним потоком.
- `ParallelBulk` - идеализированный режим, где worker threads созданы до измерения и внутри одного запуска выполняют много итераций подряд. Это верхняя оценка throughput.
- `CreateTick` - плохой runtime-режим, где потоки создаются и уничтожаются на каждый batch/tick.
- `PoolTick` - практический runtime-режим: постоянный пул потоков создан заранее, на каждый tick только сигнал start/wait done.

Вывод по batch: постоянный пул потоков нужен. Создавать потоки на каждый runtime tick нельзя: на 8192/batch 16 пул быстрее `CreateTick` примерно в `5.48x`, на 16384/batch 8 - примерно в `7.75x`. Для 128 спектров разница меньше, потому что сама работа начинает доминировать над накладными расходами, но пул все равно быстрее: `1.47x` для 8192 и `1.18x` для 16384.

Практический результат для 128 спектров:

- FFT 8192: `PoolTick` считает пачку примерно за `3.738 ms`, около `34246 spectra/s`.
- FFT 16384: `PoolTick` считает пачку примерно за `9.216 ms`, около `13888 spectra/s`.

Масштабирование не равно 16x из-за стоимости памяти, кэша, распределения работы и синхронизации потоков. Для реального движка нужно создавать worker pool при инициализации спектрального модуля и держать его живым до завершения работы.

## Выводы текущего этапа

1. Все проверенные размеры FFT от 256 до 8192 совпали с DFT-эталоном по RMS-амплитудам с запасом относительно допуска `1.0e-8`.
2. Все рабочие входные и комплексные выходные буферы реально выровнены по 64 байтам: в логе `AlignmentIn=0`, `AlignmentChampion=0`, `AlignmentSSE=0`, `AlignmentAVX=0`, `AlignmentPre=0`, `AlignmentPreAVX=0`.
3. Главный выигрыш дал не SIMD RMS, а precomputed FFT-план: удаление `SinCos` из горячего цикла butterfly.
4. `FFT_Precomputed_Pascal` быстрее старого чемпиона примерно в `4.94x..10.20x`, в зависимости от размера.
5. `FFT_Precomputed_AVX_RMS` стал победителем на 256, 512, 2048, 4096 и 8192 точках.
6. На 1024 точках победил `FFT_Precomputed_Pascal`: AVX RMS не окупился на этом размере в данном прогоне.
7. Максимальный наблюдаемый выигрыш: 256 точек, `28.160 us` -> `2.560 us`, примерно `11x`.
8. На крупном размере 8192 точек выигрыш составил `1617.920 us` -> `225.280 us`, примерно `7.18x`.

## Ограничения текущего сравнения

- SIMD backend `FFT_Iterative_SSE3_RMS`/`FFT_Iterative_AVX_RMS` ускоряет только расчет RMS-амплитуд после старого FFT.
- Precomputed backend пока остается Pascal FFT; SIMD применяется только к RMS-части.
- Еще не подключены старые реализации из `uHardwareMath.pas`, `uFFT.pas`, `FFT_койнов` и полноценные SIMD FFT-ядра.
- Замер сейчас использует среднее время (`AvgUs`), без `min/median/p95`.
- На больших размерах DFT используется только для correctness, не для полноценного speedup.
