# Правила разработки RecorderLnx

## 1. Главный стиль и согласование перед кодом
### Согласование перед реализацией
Маркер правила: `RLNX_COLLAB_BEFORE_CODE`.
Перед реализацией любого нового функционала сначала задавать пользователю наводящие вопросы и фиксировать решение. Код писать только после согласования поведения, границ и ожидаемого примера/теста.
Минимальный набор вопросов перед созданием класса или крупного блока:
- Какую задачу решает функционал и кто его будет вызывать;
- Какие входы, выходы и ошибки считаются нормальными;
- В каком потоке работает код и кто владеет данными;
- Какие существующие классы/интерфейсы нужно переиспользовать;
- Какой пример использования будет одновременно тестом;
- Что записать в карту классов после реализации.

Если пользователь уже дал точное решение, уточняющие вопросы можно сократить, но шаг согласования не пропускать.
Пишем системно и неспешно. Каждый класс или крупный блок проходит цикл: назначение и место в архитектуре, публичный интерфейс, минимальная реализация, пример использования, запись в карту классов/интерфейсов, короткая заметка в `cach/plan.md` или проектной документации.
Если функционал уже есть в проекте, используем его. Если кажется, что нужно написать похожий класс, сначала проверяем существующие исходники.

### Порядок разработки функционала
Маркер правила: `RLNX_CORE_FIRST_FLOW_2026_05_25`.
Когда возникает выбор, что делать следующим, использовать постоянную последовательность RecorderLnx:
1. Сначала ядро: доменные типы, состояния, алгоритмы, потоки, структуры данных без UI.
2. Затем минимальный интерфейс наблюдения состояния, если он помогает видеть работу ядра: индикатор состояния, журнал событий, простые команды запуска/остановки без полноценной настройки.
3. Потом инструменты настройки и тестовые функции-примеры: один или несколько примеров для проверки разных режимов и демонстрации использования.
4. Потом сохранение/загрузка конфигурации для уже проверенного ядра.
5. Только после этого полноценный визуальный интерфейс настройки, который подключается к готовому ядру и конфигурации.

UI не должен становиться местом, где впервые появляется доменная логика. Минимальный интерфейс наблюдения состояния допускается рано, но он только показывает состояние и дергает готовые команды ядра. Если для UI настройки не хватает модели или конфигурации, сначала возвращаемся к ядру/примеру/сериализации.

### Тип теста перед реализацией
Маркер правила: `RLNX_TEST_TYPE_AGREEMENT_2026_05_25`.
Перед созданием нового теста/примера в RecorderLnx сначала спрашивать пользователя, какой тип проверки нужен: отдельный консольный `.lpi`, тест внутри основного проекта, compile-time включаемый/отключаемый блок, UI-demo или другой вариант. По умолчанию для ранних UI/модельных функций предпочтительнее проверка в рамках основного проекта, если пользователь не просит отдельный тестовый проект.

## 2. Безопасное обновление и запись файлов
### Безопасное обновление файлов
Маркер правила: `RLNX_FILE_UPDATE_SAFETY_2026_05_25`.
При работе с Lazarus-проектом нельзя полагаться только на факт успешного патча или сборки: IDE может создавать `backup\`, а при копировании из backup можно случайно оставить старое время изменения или перетереть новую версию старой.
Правила обновления исходников:
- Редактировать только рабочий файл проекта, а не `backup\`, `lib\`, `__history\` или сгенерированные копии;
- После правки проверять рабочий файл по полному пути: наличие ключевых строк/методов, размер и `LastWriteTime`;
- Если восстановление делалось из backup или копированием, обновить `LastWriteTime` рабочего файла текущим временем, чтобы Lazarus предложил перечитать файл;
- Перед финальным ответом для измененных units выполнять принудительную сборку (`lazbuild -B` или `fpc -B` для теста), когда есть риск устаревших `.ppu`/`.o`;
- При подозрении на дубликаты искать файл по всему дереву проекта и сверять рабочую версию с backup;
- Не считать backup источником истины: источник истины — рабочий файл в `D:\works\OburecGH\Lazarus\RecorderLnx\...` плюс проходящий тест/сборка.

Если Lazarus не предлагает перечитать файл, первым делом проверять `LastWriteTime`, наличие дубликатов и принудительно компилировать с `-B`.

### Безопасное редактирование и патчи
- Отдавать предпочтение точечным патчам перед полной перезаписью файлов.
- Если инструмент применения патчей не может прочитать файл из-за проблем с кодировкой, не перезаписывать его вслепую. Сначала проверить текущую кодировку байт, а затем выполнить явное редактирование с сохранением кодировки или осознанную полную конвертацию.
- Если UI-исключение указывает на поле управляющего элемента (control field), убедиться, что это поле действительно создано перед использованием. Динамически создаваемые формы LCL не получают дизайнерскую инициализацию для необъявленных компонентов.

### Защита от поломки файлов при восстановлении и скриптовой записи
Маркер правила: `RLNX_SAFE_SCRIPTED_WRITES_2026_06_05`.
При восстановлении, массовой правке или редактировании файлов через скрипт нельзя писать результат сразу в рабочий исходник без промежуточной проверки. Это особенно критично для Pascal/Lazarus-файлов с CP1251/UTF-8 и для команд вида `git show ... > file`, которые при ошибке могут оставить пустой или перекодированный файл.
Правила:
- Перед записью проверить текущий размер файла и focused `git diff`;
- Восстановление или генерацию сначала писать во временный файл;
- Временный файл проверять на ненулевой размер, ожидаемые ключевые строки и корректную кодировку;
- Только после проверки заменять рабочий файл;
- После замены сразу смотреть `git diff --stat` и focused diff;
- Если diff показывает удаление всего файла, mojibake/битую кириллицу или несвязанные изменения, немедленно остановиться и восстановить файл до дальнейших правок.

## 3. Кодировки исходных файлов Lazarus / Delphi
### Стандарты кодировок
- Все `.pas` и текстовые `.dfm` в Delphi-проектах сохранять в `Windows-1251` (CP1251), без `BOM`, с переводами строк `CRLF`.
- Русские комментарии в Delphi-модулях разрешены и предпочтительны.
- Перед редактированием любого `*.pas` or `*.lfm` (Lazarus) файла с русским текстом проверять реальную кодировку файла и Pascal-директиву в верхней части модуля:
  - Байт-код файла и директива должны совпадать: `{$codepage UTF8}` для UTF-8 файлов, `{$codepage cp1251}` для Windows-1251 файлов.
- Не перезаписывать модули Pascal через дефолтные текстовые писатели без явного указания кодировки. Ошибки кодировки могут скомпилироваться успешно, но превратить заголовки LCL и ресурсы в `????`.
- Для UI-модулей Lazarus в кодировке UTF-8: сохранять как UTF-8 и использовать `{$codepage UTF8}`. Строки для LCL — обычные литералы, **без** `CP1251ToUTF8` (см. `Docs/source-encoding.md`, пилот: `UI/uTagSettingsDialog.pas`).
- Для устаревших модулей в cp1251: либо сохранять cp1251 без изменений, либо осознанно конвертировать весь модуль в UTF-8 и обновлять директиву в рамках одной правки (предпочтительный путь для RecorderLnx UI).
- Правило `CP1251ToUTF8` для cp1251-модулей остаётся до их миграции; новые и переведённые модули на UTF-8 этой обёртки не используют.
- После любых автоматических правок файлов `.pas/.dfm` проверять, что русские комментарии читаются именно как cp1251 (или UTF-8 для LCL), а не превратились в `????` или mojibake.
- После правок проверять:
  - файл открывается как обычный русский текст;
  - комментарии читаемы;
  - нет лишних пустых строк между строками кода;
  - нет BOM;
  - формат строк CRLF.

## 4. Стандарты кодирования и именования
### Именование переменных и типов
- Локальные переменные: префикс `l` (например, `lName`, `lNum`).
- Глобальные переменные: префикс `g_` (например, `g_DBase`), использовать только при крайней необходимости.
- Поля классов защищенные и приватные: префикс `f` (например, `fProperty`, `fName`).
- Члены классов с доступом: префикс `m_`.
- Интерфейсы: префикс `I` (например, `IRecorderCore`).
- Классы: префикс `T` (например, `TRecorderCore`).

### Lazarus/FPC стандарты
- Режим компиляции по умолчанию: `{$mode objfpc}{$H+}`.
- Не использовать API конкретной ОС в ядре без абстракции/интерфейса.
- UI строится исключительно на LCL.
- Бизнес-логику выносить из форм в отдельные модули (units).
- При создании/изменении форм Lazarus проектировать их визуально через формы `.lfm`, а не создавать элементы интерфейса динамически в коде. Это упрощает редактирование и локализацию.

### Документация в коде
Маркер правила: `RLNX_CODE_DOCS_2026_05_25`.
Каждый значимый модуль RecorderLnx должен содержать:
- Аннотацию в начале модуля: назначение, место в архитектуре, ограничения.
- Описание назначения публичных классов, функций и процедур.
- Описание параметров публичных функций/процедур.
- Комментарии внутри кода для сложных алгоритмов, переходов состояний и потоков.
Комментарии должны объяснять намерения разработчика, а не просто пересказывать код.

## 5. Потоки, память и производительность
### Потоки и данные
- UI-поток не должен выполнять сбор данных, запись файлов, сетевой обмен и тяжелую обработку.
- Рабочие потоки публикуют снимки данных (snapshots) или используют потокобезопасные очереди, а не напрямую передают изменяемые UI-объекты.
- Состояние рабочих потоков (например, play/suspend) менять только из самих потоков! Извне можно давать только команды.
- Не изменять существующие вызовы suspend в потоках, если они используются осознанно.

### Управление памятью
- В критических путях выполнения, циклах и рендеринге сводить к минимуму частые аллокации памяти (создание временных массивов или объектов).
- В RunTime (в потоках) память динамически не выделять! Вся рабочая память под буферы должна выделяться заранее на основе априорных данных.
- Очищать объекты вместо пересоздания: при смене состояний, запуске/останове использовать методы очистки (`Clear`), а не уничтожать и создавать объекты заново, чтобы не нарушать внешние ссылки на них.
- Для больших файлов данных (сигналы, осциллограммы) использовать постраничное чтение или Memory Mapping.

### Производительность и рефакторинг
Маркер правила: `RLNX_PERFORMANCE_ACCELERATION_2026_05_25`.
- Избегать поэлементных циклов там, где можно применить блочные высокопроизводительные функции (`Move`, `BlockRead`, специализированные математические методы).
- По возможности использовать аппаратно-ускоренные библиотеки (например, `uHardwareMath` в Lazarus SharedUtils).
- Проектировать многопоточный код без возможности дедлоков (deadlocks): минимизировать время нахождения в критических секциях, соблюдать строгий порядок захвата блокировок, избегать вложенных блокировок.
- Предотвращение избыточной перерисовки UI: UI должен перерисовываться по таймеру с фиксированной частотой (например, 5 Гц / 200 мс), а не инициироваться непосредственно при каждом приеме данных.
- Оптимизация рендеринга и отложенная сборка данных (Lazy Recompile): в графических подсистемах (например, OpenGL с пакетной отрисовкой) выполнять подготовку и сборку массивов данных только тогда, когда это действительно необходимо:
  - Выставлять инвалидирующий флаг (например, `needrecompile`) при изменении условий отрисовки: поступление новых данных, изменение масштаба отображения или изменение настроек шкалы.
  - Сборку данных и компиляцию графических структур производить непосредственно перед выводом кадра только при установленном флаге `needrecompile`, сбрасывая его по окончании сборки. Если флаг сброшен, повторно использовать ранее подготовленные структуры без перерасчета.

## 6. Доменная логика и UI
### Теги как язык обмена данными
Маркер правила: `RLNX_TAG_DATA_FLOW_2026_05_25`.
Стандартный путь данных в RecorderLnx: устройства и плагины пишут значения в теги, компоненты формуляров получают обновления через события Recorder/notify и отображают теги по своей внутренней логике. Формы не должны становиться источником доменных данных и не должны содержать функции вида `InitPlaceholderData` с реальной логикой наполнения.
Временные демонстрационные данные для UI, mock-источники и функции-примеры размещаются в `D:\works\OburecGH\Lazarus\Tests\RecorderTests\...`. Если приложение временно подключает такой модуль для раннего UI-каркаса, в шапке unit должно быть явно написано, что это тестовая заглушка до появления core-модели тегов и notify.

### Отображение тревог (Alarms)
Маркер правила: `RLNX_ALARMS_COLOR_DISPLAY`.
Срабатывание любых тревог (alarms) на каналах или системных событиях должно обязательно визуализироваться цветами в любых графических и табличных компонентах пользовательского интерфейса (например, подсветка ячеек формуляров, изменение цвета графиков/осей, мигание индикаторов состояния).

### Кроссплатформенная плагинная система
Маркер правила: `RLNX_CROSSPLATFORM_PLUGINS`.
Для реализации плагинов расширения к RecorderLnx, работающих и собирающихся под Windows и Linux по одной кнопке без изменения кодовой базы проектов, используется технология **динамических библиотек (Shared Libraries / DLL)** с плоским интерфейсом функций **Flat C API** (использование `stdcall`/`cdecl` вызовов, простых типов вроде `Double`, `Integer`, `PChar`/`PAnsiChar` в UTF-8/CP1251 и `record`-структур без динамических массивов LCL). Загрузка плагинов на стороне RecorderLnx осуществляется динамически через функции модуля `DynLibs` (`SafeLoadLibrary`, `GetProcedureAddress`, `FreeLibrary`).

## 7. Тестирование и примеры
- Для каждого нового класса необходим тест-пример: консольный тест для не-UI классов, небольшая LCL-форма для UI-компонентов, mock-источник для потоков, golden-file или проверка структуры для сериализации.
- Тест должен выполнять 2 задачи: тест корректности работы и пример использования кода.
- Для новых консольных тестов выводить наглядный лог результата: что создано, какие ключевые события произошли, какие значения попали в буфер/registry/config, и итоговую строку `RESULT ... passed`. Лог должен писаться в консоль и в `.log` файл рядом с `.exe`.
