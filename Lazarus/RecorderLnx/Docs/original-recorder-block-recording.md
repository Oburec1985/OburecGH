# Запись блоков данных в оригинальном Recorder

Документ фиксирует, как устроена запись в оригинальном Recorder из `D:\works\windev-v3.9\..`, и какие правила из этой реализации нужно сохранить в RecorderLnx. При поиске оригинальной реализации в первую очередь смотреть каталоги `rc_*` и `mr`.

## Каталог кадра

Корень записи по умолчанию в оригинальном Recorder задается как `USML` под системным корнем. Активный замер всегда пишется не в сам корень, а в отдельный каталог кадра:

```text
<DataFolder>\<FrameName>\
```

Перед стартом записи Recorder проверяет, что выбранный каталог кадра свободен или пуст. Если каталог уже есть и содержит данные, вызывается автоинкремент имени кадра. Поэтому после закрытия `D:\USML\0001\` следующий ожидаемый каталог записи должен стать `D:\USML\0002\`.

В RecorderLnx заголовок окна должен показывать полный путь именно к каталогу текущего или следующего замера, а не только корень `D:\USML\`.

## Жизненный цикл записи

Основной порядок в оригинальном Recorder:

1. Подготовка кадра: выбирается свободный каталог, создается папка, очищается состояние целей записи.
2. Подготовка целей: для каждого записываемого тега создается `CRecTarget`.
3. Создание MERA-описания: пишется главный `.mera`-файл кадра и описания персональных кадров.
4. Старт записи: цели переводятся в активное состояние.
5. Прием блоков: каждый входной блок передается в цель записи соответствующего тега.
6. Останов: цели закрывают файлы, сбрасывают буферы и освобождают дескрипторы.
7. После останова следующий кадр выбирается автоинкрементом.

## `CRecTarget`

`CRecTarget` - это дисковая цель записи одного тега. Она получает входные `FRAME`-блоки и пишет не оценку тега, а исходное содержимое блока в файлы MERA.

Для обычного векторного канала пишется сырой payload блока в:

```text
<Frame>\<TagName>.dat
```

Если у канала есть отдельная временная ось, время пишется в:

```text
<Frame>\<TagName>.x
```

Поддержанные варианты в оригинальной реализации:

- обычные векторные данные: блок пишется напрямую в `.dat`;
- скалярные теги: значение пишется в `.dat`, время устройства пишется в `.x`;
- `VT_UTS`: данные разделяются на время и значение, время идет в `.x`, значение в `.dat`;
- `VT_PAIR`: пары XY раскладываются в `.x` и `.dat`;
- packet-режим: пишется `PacketLength()` байт;
- запись с тарировкой: для нетарированных исходных типов может писаться пересчитанный `float`.

Это принципиально отличается от записи только текущей оценки или последнего значения. Для совпадения с виртуальным источником RecorderLnx должен писать полный поток блоков, включая 1D/2D данные, скаляры, разрывы и неодинаковую частоту прихода блоков.

## Аппаратная ГХ и калибровки

В оригинальном Recorder надо разделять два разных слоя:

- значения, которые попадают в `.dat`;
- описание градуировок/калибровок, которое попадает в `.mera`.

Поток значений может уже приходить после аппаратной ГХ. То есть `.dat` содержит не обязательно сырые коды АЦП, а значения, уже прошедшие аппаратное преобразование канала. При этом канальные градуировки сохраняются отдельно в `.mera`, чтобы их можно было увидеть и при необходимости подменить при последующей обработке.

В исходниках это видно по двум местам.

`CTarerTarget` в `taretrgt.cpp` подготавливает `pTaredData`: если у тега есть включенная ГХ/тарировка, данные пересчитываются через `ITag::Eval`; если нет, в tared-буфер кладется значение без пересчета. `CRecTarget` при включенном режиме `WriteTared` пишет `pFrame->TaredData()` как `float` для нескалярных каналов.

Описание канальных ГХ пишет не сам `CRecTarget`, а слой формирования `.mera`:

- `TagMeraInfoWritter.cpp` вызывает `m_pCore->WriteTaresInfo(m_file, m_pTag)` внутри секции тега;
- `CRcCore::WriteTaresInfo` в `Rc_core.cpp` перебирает `TAGPROP_TARES_COUNT` и пишет строки вида `tare0="..."`, `tare1="..."` из `TAGPROPEX_TARE_BY_INDEX`;
- `CMeraFileInfoWriter::WriteTagTareInfo` в `MeraFileInfoWriter.h` посылает тегу уведомление `TN_WRITETAGTXINFOFILE`, передавая файл `.mera`, путь кадра и имя тега. Через это уведомление тег дописывает подробную информацию о своих ГХ/калибровках в MERA-описание.

Важное правило оригинального Recorder: если запись ведется уже в оттарированном виде (`WriteTared`), `CRcCore::WriteTaresInfo` не пишет ГХ в `.mera`, чтобы не применить ту же градуировку повторно при чтении. В обычном режиме, когда в `.dat` остаются значения до программной тарировки, ссылки и описание ГХ должны быть в `.mera`.

Для RecorderLnx это означает:

1. Нельзя ограничиваться `YFormat`, `XFormat`, частотой и единицами. Секция каждого измерительного тега должна уметь сохранять список канальных ГХ/калибровок.
2. Данные, уже прошедшие аппаратную ГХ, пишутся как значения канала, но датчиковая и усилительная градуировки должны сохраняться отдельно в `.mera` как метаданные канала.
3. При реализации block-recording нужно явно фиксировать режим: пишем коды, аппаратно приведенные значения или программно оттарированные значения. От этого зависит, какие ГХ пишутся в `.mera`, чтобы чтение не применило преобразование второй раз.

## Потоки и производительность

Оригинальный Recorder не делает тяжелую файловую работу из UI-сценария. Поток поступления данных передает блоки в цели записи, а `CRecTarget` держит собственные файловые дескрипторы и защищает переключение файлов критической секцией.

Важные свойства:

- файлы открываются заранее на этапе подготовки;
- данные дописываются блоками, без пересборки всего файла;
- для чтения во время записи используется режим совместного доступа;
- часть файлов заранее резервируется на диске;
- критическая секция защищает только короткий участок переключения/записи;
- периодический flush применяется особенно для скалярных/временных данных, чтобы уже записанная часть переживала падение процесса.

Для RecorderLnx это означает: acquisition/data-source поток не должен выполнять долгий disk I/O. Нужна очередь блоков или эквивалентный неблокирующий слой, а запись на диск должна идти отдельным writer-контуром с короткими блокировками.

## Предыстория

Предыстория в оригинальном Recorder пишется не в память, а прямо на диск через `CCycleFile`.

Для каждого сигнала создаются две временные пары файлов:

```text
<TagName>_1
<TagName>_2
<TagName>.x_1
<TagName>.x_2
```

При просмотре, пока запись еще не начата, входные блоки попадают в кольцевой файл. Когда лимит глубины предыстории достигнут, активный файл переключается, старый перематывается и используется повторно.

При старте записи:

- активная часть кольцевого файла становится началом итогового `.dat`;
- активная временная часть становится началом итогового `.x`;
- неактивные временные файлы удаляются;
- дальше запись продолжается уже в итоговые файлы кадра.

Так Recorder добавляет к ручному или триггерному старту уже накопленную предысторию без большой операции копирования из памяти.

## Паузы и разрывы данных

Оригинальный Recorder поддерживает запись с паузами через режим `RecordingWithPause` / `RCPROP_ENABLEDRECPAUSE`. При паузе запись не закрывает весь кадр как законченный замер, а фиксирует границу новой порции данных для каждого измерительного тега.

В коде оригинала это устроено так:

- `CRcCore::Pause` вызывает `StopRec` у тегов, затем `WriteBreakInfo`, после этого `UpdatePortionInfoFile`;
- `CRcCore::UpdatePortionInfoFile` вызывает `CMeasurementTag::UpdatePortionsForPauseMode`;
- `UpdatePortionsForPauseMode` добавляет в `m_portions` пару `(offset, time)`, где `offset` - смещение начала новой порции в уже записанном `.dat`, а `time` - время начала этой порции;
- `CMeasurementTag::WriteServiceInfo` пишет service-файл порций рядом с данными через `WritePortionsInfo(zsPath + ".prt")`;
- `CPortionsInfo::Write` сохраняет файл текстовыми строками:

```text
<time> <offset>
```

В этом дереве оригинального Recorder расширение service-файла порций - `.prt`. В некоторых описаниях его легко перепутать с `.ptr`; при реализации RecorderLnx надо свериться с ожидающим читателем MERA/Winpos. Смысл файла: хранить точки разрыва/продолжения временной оси, чтобы один `.dat` мог состоять из нескольких временных порций.

Кроме ручных пауз есть автоматическая фиксация разрывов по потере блоков. `CMeasurementTag` проверяет интервалы между блоками для каналов без собственной временной оси. Если следующий блок пришел позже ожидаемого больше чем на половину длительности блока, в `m_portions` добавляется новая граница порции, а Recorder поднимает состояние ошибки приема. Комментарий в исходнике отдельно отмечает, что XY-каналы сами способны передать разрывы в данных; для `VT_UTS`, `VT_PAIR` и скалярных тегов время хранится явно через `.x`.

Для RecorderLnx это означает:

1. Пауза записи не должна просто “ничего не писать” и потом продолжать тот же равномерный ряд без метаданных.
2. Для каналов без собственной X-оси writer должен вести service-файл порций (`.prt` в изученной версии оригинала; проверить совместимость с `.ptr` в целевом Winpos).
3. Каждая запись в service-файл должна связывать смещение в `.dat` с реальным временем начала новой порции.
4. Автоматические разрывы из-за пропуска блоков должны фиксироваться тем же механизмом порций.
5. Для каналов с явным временем (`.x`, `VT_UTS`, `VT_PAIR`, скаляры) разрывы должны быть представлены в самой временной оси и валидироваться отдельно.

## MERA-описание

Главный `.mera`-файл создается в каталоге кадра. В оригинале он содержит:

- секцию `[MERA]`;
- приложение-источник и версию;
- дату/время старта кадра;
- секции тегов;
- частоту;
- единицы X/Y;
- комментарии, адреса и аппаратные параметры;
- формат Y-данных;
- формат X-данных для временных/скалярных/XY-каналов;
- ссылки и описания канальных ГХ/калибровок;
- service-информацию о порциях записи/разрывах для каналов без собственной X-оси;
- служебные и пользовательские свойства.

Для параметрических тегов описание дополнительно связывает X/Y-файлы. Для персональных кадров группы получают уведомления подготовки и создания info-файлов.

## Правила для RecorderLnx

1. Каталог записи в UI и логике - это полный каталог кадра `RootDir + FrameName`, например `D:\USML\0001\`.
2. После закрытия кадра следующий отображаемый путь должен стать следующим свободным номером, например `D:\USML\0002\`.
3. Запись MERA не должна строиться из UI-снимков и оценок тегов. Источником записи должен быть полный поток блоков источника.
4. Для каждого записываемого канала нужен writer, который знает тип данных канала и пишет `.dat`/`.x` в формате, совместимом с MERA.
5. Потоки источников не должны тормозить на диске: они передают неизменяемые block-пакеты в очередь записи.
6. Writer должен дописывать данные инкрементально и периодически сбрасывать буферы, чтобы при падении процесса на диске оставалось все, что успело пройти через writer.
7. Предыстория должна вестись дисковым кольцевым буфером, а не только памятью.
8. `.mera` должен сохранять канальные датчиковые и усилительные ГХ/калибровки отдельно от значений `.dat`.
9. Запись с паузами и потерями блоков должна сохранять границы порций в service-файле (`.prt` в оригинале; проверить ожидаемое `.ptr` для целевого читателя).
10. Валидация записи должна сравнивать результат с виртуальным источником поблочно: `.dat`, `.x`, service-файл порций, количество отсчетов, времена, пропуски, типы данных и наличие ГХ в `.mera`.

## Текущий разрыв RecorderLnx

Текущая реализация `TRecorderMeraTagWriter.WriteSample` и вызовы из `TMainForm.ApplyTagEventSnapshot` пишут семплы из снимков тегов. Этого достаточно для грубой записи текущих оценок, но недостаточно для полной блочной записи исходного потока. Для совместимости с оригинальным Recorder нужен отдельный слой block-recording, подключенный к публикации блоков источников до агрегации в оценки тегов.
