# Правила приёма данных MIC-140

Зафиксировано по итогам обсуждения с разработчиком оригинального Recorder.
Обязательны для **MIC140v2**; текущий `Device/MIC140` частично совпадает (см. таблицу в конце).

---

## 1. Поиск начала кадра MDP

При разборе TCP-потока, если синхрослово или контрольные суммы не сходятся, сдвиг
**на один байт** и повторный поиск `0x12B8`. Нельзя отбрасывать весь «подозрительный»
кусок целиком — иначе теряется выравнивание, прибор продолжает слать скан, а ПО
уходит в бесконечный resync.

**Код:** `TRecorderMic140LegacyClient.ReadPacket` — `DropRxBytes(1, True)`.

---

## 2. Заголовок верный, данные «мусор»

Если MDP-заголовок и `THeaderMessage` выглядят правильно, а отсчёты АЦП — шум,
скачки на ±32767, несогласованные `num_buff` — **чаще всего ошибка настройки**:
циклограмма коммутатора, тайминг переключения каналов, размер FIFO, stride каналов.
Это не «битый Ethernet», а неверная программная конфигурация скана.

**Действия ПО:**

- помечать блок как повреждённый (`Mic140LegacyRawBlockLooksCorrupt`);
- не публиковать в теги;
- считать `corruptRead`, при серии ошибок — мягкий перезапуск скана;
- в логе явно указывать: проверить программирование, а не только resync TCP.

---

## 3. Кольцо заранее выделенных слотов

Поток **чтения** пишет каждый принятый кадр в **следующий слот** фиксированного
кольца (`TMic140LegacyRawBlock` с заранее известным числом слов). Поток **разбора**
берёт другой слот. Индексы чтения/записи чередуются — потоки не делят один буфер
и не держат общую критическую секцию на время копирования сотен слов.

Требования:

| Требование | Смысл |
|------------|--------|
| Слоты выделены при старте | Без `SetLength` на горячем пути |
| Запись = копирование в слот `writeIndex` | Поток чтения не трогает слот, который разбирает publish |
| Короткая блокировка | Только сдвиг индексов / счётчик глубины |
| Переполнение | Сброс старейшего + предупреждение в лог |

**Целевой модуль:** `uRecorderMic140RawRing.pas` (8 слотов по умолчанию).

**Сейчас в legacy:** `TMic140RawDoubleBuffer` (2 слота) — временно; v2 переходит на кольцо.

---

## 4. Один интервал ожидания в режиме опроса

В состоянии **идёт опрос** поток чтения делает **один** `WaitFor` на период накопления
блока (`DataUpdateMs` / `AcquirePacingMs`), согласованный с частотой прибора и периодом
обновления UI. Дополнительные `Sleep` в этом потоке не нужны.

| Поток | Допустимое ожидание |
|-------|---------------------|
| Чтение, опрос | **Один** pacing-интервал на итерацию |
| Чтение, не опрос | `IdleWaitMs` (ожидание старта) |
| Разбор блоков | Пробуждение по событию после `Enqueue`; без опросного pacing |
| Разбор, нет данных | Ожидание события (не короткий spin-sleep 25 ms в цикле опроса) |

Тайминги и этапы — через общий базовый класс `TRecorderAcquirePhaseFsm` /
`TRecorderAcquireTimingBase` (`Device/uRecorderAcquirePhase.pas`). MIC-140 задаёт
числа в `TRecorderMic140AcquireTiming`; при необходимости — переопределение методов.

---

## Связь с кодом

| Правило | Legacy (`MIC140`) | v2 (`MIC140v2`) |
|---------|-------------------|-----------------|
| 1. Побайтовый resync | `uRecorderMic140LegacyProtocol` | Тот же transport |
| 2. Мусор при верном заголовке | `Mic140LegacyRawBlockLooksCorrupt` | Сохранить + диагностика |
| 3. Кольцо слотов | `TMic140RawDoubleBuffer` (2) | `TMic140RawBlockRing` (8+) |
| 4. Один pacing-sleep | `DataUpdateMs` в read-thread | `AcquirePacingMs` из базового FSM |
| Этапы + тайминги | `TMic140StreamFsm` + `ScanConfig` | Наследование от `TRecorderAcquirePhaseFsm` |

См. также [protocol.md](protocol.md) §4, [device_abstraction.md](../device_abstraction.md) §5.
