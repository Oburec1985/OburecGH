# MIC-140: протокол обмена (legacy MC031 / mdpEthernet81)

Краткое описание обмена по TCP для прибора на порту **4000** (исполнение 48 каналов,
подсеть `192.168.14.*`). Числа — **little-endian**.

**Код:** `Device/MIC140/uRecorderMic140LegacyProtocol.pas`,  
`Device/MIC140/uRecorderMic140DataSource.pas`.

**Эталон:** `windev-v3.9` — `mdpEthernet81`, `Mc031ethernetifc`, `Ccdevice.h`, `devapi/Types.h`.

---

## Общая оболочка MDP-кадра

Все посылки (и измерения, и команды) идут в одном TCP-потоке и имеют общий заголовок.

Первая строка таблицы — **начало кадра** в потоке байт TCP.

| ID | Формат | Размер | Назначение |
|----|--------|--------|------------|
| MDP-SYNC | `uint16` | 2 байта | **Начало кадра.** Синхрослово `0x12B8` |
| MDP-PORT | `uint16` | 2 байта | Номер потока: `0` — блоки скана, `1` — команды и ответы |
| MDP-SIZE | `uint16` | 2 байта | Длина полезной нагрузки в словах WORD |
| MDP-HCS | `uint16` | 2 байта | Контрольная сумма заголовка: `(SYNC+PORT+SIZE) mod 65536` |
| MDP-PAYLOAD | `uint16[]` | `SIZE`×2 байт | Тело кадра (см. разделы ниже) |
| MDP-DCS | `uint16` | 2 байта | **Конец кадра.** Сумма всех слов payload mod 65536 |

## 1. Соединение и поиск прибора

### 1.1. Последовательность

1. **Поиск по сети** (диалог настроек): перебор `192.168.14.1…254`, проверка доступности TCP **порта 4000** (ICMP ping не обязателен).
2. **Установка TCP-соединения** с выбранным IP.
3. **Идентификация:** команда `CMD_REPLY` (113) — см. таблицу 1.2.
4. При корректном ответе выбирается **legacy-путь** MC031; иначе пробуется Mebius (другие исполнения MIC140MDQ).

Предварительная проверка TCP до `Connect` нужна, чтобы не ловить исключение сокета при отсутствии прибора.

### 1.2. Таблица. Запрос идентификации (поток 1)

Кадр MDP с `PORT=1`. Полезная нагрузка — тело команды CC BIOS.

| ID | Формат | Размер | Назначение |
|----|--------|--------|------------|
| CMD | `uint16` | 2 байта | **Начало тела команды.** `113` = `CMD_REPLY` |
| ARGC | `uint16` | 2 байта | Число аргументов (`0`) |
| RETC | `uint16` | 2 байта | Ожидаемое число слов ответа (`11`) |

### 1.3. Таблица. Ответ идентификации (поток 1, 11 слов)

| ID | Формат | Размер | Назначение |
|----|--------|--------|------------|
| SIGNATURE | `uint16` | 2 байта | **Начало ответа.** Подпись BIOS |
| MDP-TYPE | `uint16` | 2 байта | Тип MDP |
| DEV-TYPE | `uint16` | 2 байта | Тип устройства |
| DEV-REV | `uint16` | 2 байта | Ревизия |
| DEV-SER | `uint16` | 2 байта | Заводской номер |
| CC-TYPE | `uint16` | 2 байта | Тип контроллера CC |
| CC-SER | `uint16` | 2 байта | Серийный номер CC |
| EEPROM-MFG | `uint16` | 2 байта | ID производителя EEPROM |
| EEPROM-DEV | `uint16` | 2 байта | ID устройства EEPROM |
| BIOS-FUNC | `uint16` | 2 байта | Функции BIOS |
| BIOS-VER | `uint16` | 2 байта | Версия BIOS |

После успешного ответа прибор считается **подключённым** (`rdsConnected`).

## 2. Настройка (программирование скана)

Настройка выполняется **до** запуска опроса: запись дескрипторов в DM-память ADSP,
привязка FIFO и таблицы каналов к контексту скана BIOS.

### 2.1. Последовательность команд

| Шаг | Команда                | Код | Назначение                               |
| --- | ---------------------- | --- | ---------------------------------------- |
| 1   | `CMD_RESETSCANMAIN`    | 83  | Сброс контекстов скана                   |
| 2   | `CMD_CONFIGSCANMAIN`   | 84  | Таймер главного скана (частота из сетки) |
| 3   | `CMD_APPENDSCANMAIN`   | 82  | Добавить внутренний контекст MIC-140     |
| 4   | `CMD_SETSTATESCAN`     | 87  | Состояние контекста                      |
| 5   | `CMD_WRITEDM`          | 111 | Запись дескрипторов FIFO и каналов в DM  |
| 6   | `CMD_SCAN_SET_BUFF`    | 133 | Привязка FIFO к скану                    |
| 7   | `CMD_ADDCHANNELMODULE` | 152 | Модульные каналы (MIC-140)               |
| 8   | `CMD_SCAN_SET_CHANS`   | 132 | Таблица каналов скана                    |

**Важно:** для сканирования используются коды из `Ccdevice.h` (например WRITEDM = **111**, не 16 из VTBL.H).

Память DM (Ethernet MC031): буфер скана `0x0522…0x07FF`, куча дескрипторов `0x0800…0x2BFF`. Перед новой настройкой курсоры аллокации сбрасываются.

### 2.2. Таблица. Тело команды (поток 1, общий формат)

| ID | Формат | Размер | Назначение |
|----|--------|--------|------------|
| CMD | `uint16` | 2 байта | **Начало тела.** Код команды CC BIOS |
| ARGC | `uint16` | 2 байта | Число слов в `ARGV` |
| RETC | `uint16` | 2 байта | Ожидаемое число слов ответа (`0` → минимум 1) |
| ARGV | `uint16[]` | `ARGC`×2 байт | Аргументы (адрес DM, данные и т.д.) |

Ответ — кадр MDP `PORT=1` с полезной нагрузкой из `RETC` слов (или больше).

### 2.3. Таблица. Пример `CMD_WRITEDM` (111)

| ID | Формат | Размер | Назначение |
|----|--------|--------|------------|
| CMD | `uint16` | 2 байта | `111` |
| ARGC | `uint16` | 2 байта | `2 + N` (адрес + счётчик + данные) |
| RETC | `uint16` | 2 байта | `0` |
| DM-ADDR | `uint16` | 2 байта | Адрес в DM |
| WORD-CNT | `uint16` | 2 байта | Число записываемых слов `N` |
| DATA | `uint16[]` | `N`×2 байт | Записываемые слова |

После успешной цепочки прибор **запрограммирован** (`rdsProgrammed`).

## 3. Опрос (приём блоков измерений)

### 3.1. Запуск и останов

| Действие | Команда | Код | Условие |
|----------|---------|-----|---------|
| Старт скана | `CMD_STARTSCANMAIN` | 80 | Состояние «запрограммирован» |
| Стоп скана | `CMD_STOPSCANMAIN` | 81 | Состояние «идёт опрос» |

Стоп по таймауту чтения **не** отправляется — только по явной остановке пользователем или отключению.

### 3.2. Таблица. Кадр потока 0 (блок скана, 48 каналов, 2 отсчёта)

Первая строка — начало полезной нагрузки MDP (`PORT=0`, `SIZE=106` слов).

| ID | Формат | Размер | Назначение |
|----|--------|--------|------------|
| BIOS-TYPE | `uint16` | 2 байта | **Начало BIOS-сообщения.** `0` = данные |
| BIOS-SIZE | `uint16` | 2 байта | Размер сообщения в словах (`106`) |
| BIOS-SCAN_ID | `uint16` | 2 байта | Идентификатор скана |
| BIOS-SLOT | `uint16` | 2 байта | Слот |
| BIOS-CHAN | `uint16` | 2 байта | Канал |
| BIOS-TIME_HI | `uint16` | 2 байта | Метка времени, старшая часть |
| BIOS-TIME_LO | `uint16` | 2 байта | Метка времени, младшая часть |
| BIOS-TIME_CNT | `uint16` | 2 байта | Счётчик времени BIOS |
| BIOS-NUM_BUFF | `uint16` | 2 байта | Порядковый номер блока (монотонный) |
| BIOS-STATE | `uint16` | 2 байта | Флаги состояния блока |
| DATA | `uint16[]` | 96 слов | `2 отсчёта × 48 AIn`, шаг 48; код АЦП как `int16` |
| *(MDP-DCS снаружи payload)* | `uint16` | 2 байта | **Конец MDP-кадра** |

**Итого кадр:** 8 байт MDP-заголовок + 212 байт payload + 2 байт DCS = **222 байта**.

Заголовок BIOS — **ровно 10 слов** (`THeaderMessage`). Ошибочное чтение 5 слов приводит к попаданию служебных полей в отсчёты.

Внутренний stride скана для 48+3 каналов (AIn + TIn) задаётся циклограммой; пользовательские теги получают 48 AIn, TIn обрабатываются внутри драйвера (CJC).

### 3.3. Потоки в RecorderLnx

```text
Прибор :4000
  → поток чтения: ReadLegacyRawBlock
  → кольцевой буфер сырых кадров
  → поток публикации: декоммутация → теги MIC140_*
```

Команды во время опроса не должны «съедать» кадры потока 0: при ожидании ответа команды пакеты `PORT=0` пропускаются до прихода `PORT=1`.

## 4. Восстановление обмена и служебные сообщения

### 4.1. Поиск начала кадра

При ошибке sync или контрольных сумм — сдвиг TCP-потока **на 1 байт** и повторный поиск
`0x12B8`. Подробнее: [acquisition_rules.md](acquisition_rules.md) §1.

### 4.2. Заголовок верный, отсчёты — мусор

Вероятная причина — **ошибка настройки** циклограммы/коммутатора/FIFO, а не «битый кабель».
ПО не публикует такие блоки в теги. См. [acquisition_rules.md](acquisition_rules.md) §2.

### 4.3. Отдельных push-уведомлений от прибора нет

Прибор **не** шлёт асинхронных «событий» вне потоков 0 и 1. Диагностика — по полям `BIOS-NUM_BUFF`, `BIOS-STATE` в каждом блоке скана и по кодам ошибок команд.

### 4.4. Потеря синхронизации MDP

При неверном sync, контрольной сумме заголовка или payload парсер **сдвигает поток на 1 байт** и снова ищет `0x12B8` (как `mdpEthernet81::ProcessProtocol`). Сброс всего кандидата целиком рассинхронизирует TCP при активном скане.

### 4.5. Типовые меры в RecorderLnx

| Ситуация | Действие |
|----------|----------|
| Повторная настройка после уже запущенного скана | Предварительный `CMD_STOPSCANMAIN` |
| Зависание / битые блоки | Мягкий перезапуск скана (`LegacyTrySoftRestartScan`) |
| Долгое отсутствие кадров | Перезапуск потока после stall (`LegacyTryRestartStreamAfterReadStall`) |
| Разрыв TCP | `LegacyRecoverTcpConnection`: disconnect → connect → program → start |
| Пропуски `num_buff` | Учёт в счётчиках gap/duplicate, лог диагностики |
| Очистка после stop | Сброс внутреннего RX-буфера, чтобы старые кадры потока 0 не мешали ответам команд |

### 4.6. Таблица. Поля блока скана для контроля целостности

| ID | Назначение |
|----|------------|
| BIOS-NUM_BUFF | Монотонный номер; пропуски — признак потери кадров |
| BIOS-STATE | Флаги состояния BIOS на момент выборки |
| MDP-DCS | Ошибка суммы — повод к побайтовому resync |

## Связанные состояния прибора в ПО

| Состояние | Смысл |
|-----------|--------|
| Отключён | Нет TCP / файл не открыт |
| Подключён | Идентификация пройдена, скан не настроен |
| Запрограммирован | Циклограмма и DM записаны |
| Идёт опрос | Отправлен `CMD_STARTSCANMAIN`, принимается поток 0 |

Подробности настройки каналов, TIn/CJC, частот — в [mic140_protocol.md](../../mic140_protocol.md).
