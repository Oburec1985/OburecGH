# MIC-140 quick start для разработчика RecorderLnx

Документ фиксирует рабочую последовательность программирования MIC-140,
которая была проверена на приборе `192.168.14.155:4000`, 48 каналов, 100 Гц.
Оригинальные исходники Recorder: `D:\works\windev-v3.9`.

## 1. Подключение

RecorderLnx сначала пробует legacy Ethernet протокол MC031/mdpEthernet81:

1. TCP probe на `host:4000`, чтобы не ловить исключение `TInetSocket.Create`
   как штатный timeout.
2. Создание `TRecorderMic140LegacyClient`.
3. `ReadFirmware`, команда `CMD_REPLY = 113`.
4. Если ответ корректный, источник работает через legacy protocol.
5. Если legacy не ответил, оставлен fallback на новый Mebius/MEAS протокол.

Код: `Device/MIC140/uRecorderMic140DataSource.pas`,
`TRecorderMic140Device.Connect`.

Подробнее о потоке scan и отладке: [`mic140_legacy_scan_stream.md`](mic140_legacy_scan_stream.md).

## 2. Диагностика связи

При `ConfigureTags` MIC-140 создает два служебных тега:

- `<source>_Status` - состояние подключения и сбора;
- `<source>_Blocks` - число успешно полученных scan-блоков.

Коды `Status`:

| Код | Значение |
|---:|---|
| `0` | нет связи / остановлено |
| `1` | TCP и протокол определены |
| `2` | scan запрограммирован |
| `3` | scan запущен, данные идут |
| `-1` | ошибка: connection/program/start/read |

Текстовое состояние дополнительно пишется в `TextValue` и в лог:

```text
[DataSource:...] MIC-140 status=3 started; data ok
```

Если три чтения подряд не вернули scan-блок, статус становится `-1` /
`no scan data`.

## 3. Программирование legacy scan

Рабочая последовательность повторяет старый Recorder:

1. Если предыдущий scan уже запускался, отправить `CMD_STOPSCANMAIN = 81` и
   очистить буфер принятых пакетов.
2. `CMD_RESETSCANMAIN = 83`.
3. `CMD_CONFIGSCANMAIN = 84`.
   Для 100 Гц на 16 МГц сетке:
   `scale=1`, `period=640`, в команду уходят `0, 639`.
4. Выделить scan context в DM heap `0x0800..0x2BFF`.
5. `CMD_APPENDSCANMAIN = 82`.
   Важный параметр - `scanDivider`, не индекс комбобокса частоты.
   Для 100 Гц `scanDivider=500`.
6. `CMD_SETSTATESCAN = 87`, state `0`.
7. Создать FIFO descriptor и привязать его через `CMD_SCAN_SET_BUFF = 133`.
8. Записать дескрипторы каналов и таблицу указателей.
9. Привязать каналы командами `CMD_ADDCHANNELMODULE = 152` и
   `CMD_SCAN_SET_CHANS = 132`.
10. После успешного программирования `Start` отправляет
    `CMD_STARTSCANMAIN = 80`.

### Осторожно с TIn/CJC

TIn нельзя добавлять в scan как обычные пользовательские каналы после 48 AIn.
Проверка на приборе `192.168.14.155` показала, что такой scan меняет раскладку
блока, замедляет поток и портит данные AIn.

В оригинальном Recorder TIn используются как внутренние корректоры холодного
спая:

1. AIn канал при `ConfigBuffer()` получает корректор через
   `GetDefaultCorrector()`.
2. Для 48-канального исполнения v3 таблица `TINCOR_AIN_INDEX_48V3` связывает
   группы AIn с T1..T5.
3. `UpdateCjcTables()` группирует AIn по TIn-корректору.
4. `Decommutation()` усредняет TIn за блок, сглаживает температуру коэффициентом
   `GetCjcChanK()` и применяет ее к термопарным AIn.

В RecorderLnx рабочий путь пока оставлен только на 48 AIn. Когда переносится
CJC, TIn надо программировать как внутренние корректоры с тем же порядком
описателей, что в старом Recorder. Диагностические значения TIn можно выводить
наружу только как рассчитанные скаляры за блок, а не как обычные каналы данных.

## 4. Память и FIFO

Для Ethernet MC031 начало scan FIFO не `0x0000`, а `0x0522`.

Источник: `D:\works\windev-v3.9\mtcEthernet81\Mc031ethernetifc.cpp`,
`CCMC031EthernetInterface::GetDMAddrBeginBuffer()`.

Для 48 каналов:

```text
DM FIFO range       = 0x0522..0x07FF
available words     = 734
ready per channel   = (734 / 2) / 48 = 7
ready scan words    = 7 * 48 = 336
descriptor size     = 2 * 336 = 672
descriptor sizeready= 336
```

Именно `336` является готовой порцией scan, а не `7`. Ошибка здесь приводила
к зависанию обмена после первого похожего на реальные данные блока.

## 5. Частотная сетка

В UI сейчас оставлена безопасная часть оригинальной сетки для 48 каналов:

```text
1, 2, 5, 10, 20, 25, 50, 100 Hz
```

Для 16 МГц `scale_period_16000`:

| Частота | scale | period | count |
|---:|---:|---:|---:|
| 1 | 2 | 640 | 25000 |
| 2 | 1 | 640 | 25000 |
| 5 | 1 | 640 | 10000 |
| 10 | 1 | 640 | 5000 |
| 20 | 1 | 640 | 2500 |
| 25 | 1 | 640 | 2000 |
| 50 | 1 | 640 | 1000 |
| 100 | 1 | 640 | 500 |

Старый Recorder динамически режет сетку по числу каналов, заземлению,
времени ожидания и усреднению. В RecorderLnx быстрые строки пока скрыты,
чтобы не загонять прибор в невозможную циклограмму.

## 6. Каналы

В legacy scan сейчас программируются 48 AIN каналов.

Порядок физических каналов 48-канального исполнения:

```text
24..47, затем 23..0
```

Перед каждым каналом включается дескриптор земли. Таблица указателей в BIOS
выглядит как:

```text
ground, ch1, ground, ch2, ... ground, ch48
```

Дескриптор канала состоит из 5 WORD:

```text
code_ME048[0]
code_ME048[1]
desc_MIC140
delay
value_ptr
```

Для текущего рабочего режима дескрипторы собираются в
`TRecorderMic140Device.ProgramLegacyDevice`.

## 7. Цикл сбора данных

После `CMD_STARTSCANMAIN` источник делает:

1. `DoTick` вызывает `ReadBlock`.
2. `ReadScanBlock` читает stream `0`.
3. Из scan-пакета отбрасывается `THeaderMessage = 10 WORD`.
4. Оставшиеся WORD раскладываются как кадры по 48 каналам:

```pascal
lDataIndex := J * ABlock.ChannelCount + I;
ABlock.Values[I][J] := SmallInt(lLegacyBlock.DataWords[lDataIndex]);
```

5. `DoTick` публикует блоки в теги через `Registry.PublishBlock`.
6. При успешном блоке увеличивается диагностический `<source>_Blocks`.

Оригинальный Recorder читает TCP в отдельном RX-потоке и раскладывает stream
`0` / stream `1` по двум FIFO. В RecorderLnx пока чтение синхронное, но
устойчивость уже достаточна для первого рабочего обмена на 100 Гц.

## 8. Остановка

Остановка источника:

1. Если scan был запущен, отправить `CMD_STOPSCANMAIN = 81`.
2. Очистить buffered packets.
3. Сбросить флаги чтения и `fLegacyScanWasStarted`.
4. Закрыть TCP-соединение.
5. Опубликовать диагностический статус `0 / stopped`.

Это важно: если не остановить scan перед новой программой, BIOS может остаться
в старом состоянии, а следующий запуск начнет читать хвосты старого потока.

## 9. Компенсация холодного спая

Оригинальный Recorder делает CJC после декоммутации блока:

1. `ScanMIC140::UpdateCjcTables` группирует термопарные каналы по
   каналу-компенсатору и имени тарировки термопары.
2. Для канала компенсации считается средняя температура блока.
3. Температура сглаживается:

```text
lastT = T * k + (1 - k) * lastT
```

4. Для каждого термопарного канала берется `Temper = lastT + temper_offset`.
5. Через тарировку термопары считается эквивалентная ЭДС холодного спая:
   `Temper_mV = tare.EvalInverse(Temper)`.
6. К текущей ЭДС термопары добавляется `Temper_mV`.
7. Итог переводится обратно в температуру: `tare.Eval(value + Temper_mV)`.

В RecorderLnx UI уже содержит настройки CJC и температурные каналы, но полный
legacy scan температурных каналов и слой тарировок `Eval/EvalInverse` еще
нужно перенести перед включением физической CJC. Не включать CJC "на глаз":
это меняет таблицу BIOS-каналов, размер кадра и порядок данных.

Минимальный безопасный следующий шаг для CJC:

1. Перенести дескрипторы `TIn` из `MIC140_48mod.cpp`.
2. Расширить legacy scan до `48 AIN + выбранные TIn`.
3. Научить `ReadBlock` читать фактический frame size, а публиковать только
   выбранные теги.
4. Перенести или реализовать тарировочный интерфейс `Eval/EvalInverse`.
5. Только после этого включить `ThermoCompensationEnabled` в настройках.
