# RecorderLnx: идеи из руководства пользователя Recorder

Источник: `D:\works\AGrav\temp\МЕРА\Recorder\Recorder. Руководство пользователя.pdf`, редакция 3.2, 2023.

## Назначение Recorder

Recorder работает в составе измерительных комплексов MIC. Для RecorderLnx берем не привязку к конкретному Windows-стеку, а рабочую модель:

- ввод оцифрованных аналоговых сигналов из измерительных модулей;
- сохранение сигналов на долговременный носитель;
- автоматическое определение состава аппаратуры: типы и расположение модулей, плат, внешних модулей;
- настройка модулей, плат и измерительных каналов;
- калибровка/градуировка каналов;
- просмотр сигналов в темпе эксперимента в графическом и цифровом виде;
- экспресс-обработка и плагины в темпе измерений.

## Термины, которые переносим в модель

- Замер/кадр: однократная реализация физического процесса, записываемая в MERA или USML. В RecorderLnx это будущая сущность `MeasurementRun` или `Frame`.
- Рабочий каталог: каталог, содержащий замеры.
- Измерительный тракт канала: цепочка передачи и преобразования сигнала, образующая измерительный канал.
- ГХ: градуировочная характеристика логического канала для перевода электрических величин в физические.
- КХ: аппаратная калибровочная характеристика для преобразования кодов АЦП в электрическую величину.
- Тахо-канал: аналоговый или цифровой канал базовой частоты агрегата.
- Триггер: набор параметров сигнала для выделения момента изменения логического состояния.
- Уставка: именованный триггер, обычно назначенный на тренд оценки; при срабатывании выполняется действие, например предупреждение.
- Формуляр отображения: окно, отображающее сигнал/сигналы или параметры в графическом или цифровом виде.
- Плагин: дополнительный программный модуль, расширяющий Recorder.

## Рабочие режимы

Руководство фактически разделяет:

- Stop/Остановлен: настройка конфигурации, аппаратуры, каналов, формуляров, путей записи.
- Preview/Просмотр: запуск F3, проверка системы измерений и каналов перед записью, отображение входных сигналов в реальном времени без записи на диск.
- Record/Запись: запуск F2, запись в выбранный кадр/каталог, отображение данных в темпе эксперимента.

Важная деталь для `TRecorderStateMachine`: режим может быть выбран пользователем, но фактический старт просмотра/регистрации может ждать условия старта. Условия старта/остановки из руководства:

- по клавише;
- по уровню сигнала с выбором канала, порога и направления фронта;
- по внешнему управляющему сигналу на цифровой вход, то есть TTL/trigger start;
- остановка по клавише ESC;
- остановка по уровню;
- остановка через заданное время.

Следствие: первый state machine должен либо иметь отдельное состояние `WaitingForStartTrigger`, либо хранить фазу `Armed` внутри Preview/Record. Это надо согласовать перед кодом.

## UI-модель

Основное рабочее окно Recorder имеет три главные панели:

- панель сигналов;
- панель управления;
- панель списка каналов.

В рабочем окне должны быть видны:

- текущий режим и длительность текущего просмотра/записи;
- имя текущей конфигурации;
- полное имя кадра/записи;
- список подключенных/рабочих каналов с названием и местом установки;
- формуляры отображения: осциллограммы, цифровые значения, тренды, гистограммы, мнемосхемы.

В режиме просмотра активный канал можно менять во время работы. Если каналов больше, чем окон осциллограмм, отображается часть сигналов начиная с активного канала.

Расширенный UI поддерживает страницы мнемосхем: создание страниц, добавление элементов, перемещение/изменение размера, выравнивание, фон страницы, преднастроенные компоненты, цифровые индикаторы, текстовые метки, окна осциллограмм и трендов.

## Конфигурация

Из руководства для первой архитектуры важны:

- загрузка и сохранение конфигурации;
- быстрый старт через выбор доступных/выбранных каналов;
- настройки Recorder: путь регистрации данных, шаблон имени кадра, модификация имени по каждому испытанию;
- запуск с параметром `/cfg:<имя файла проекта>` в Windows Recorder; для RecorderLnx нужен кроссплатформенный аналог CLI-параметра;
- аппаратные свойства: дерево/список устройств, поиск и добавление модулей, настройка модулей, плат, Ethernet-контроллеров;
- настройка измерительных каналов отдельно от физических модулей.

## Данные и форматы

Руководство описывает USML и MERA:

- USML: один информационный файл `*.usm` с заголовком, таблицей паспортов параметров и массивами значений.
- MERA/USMLext: набор файлов в каталоге; `*.mera` как ini-подобный описатель испытания и параметров, `*.dat` для двоичных данных параметра, `*.x` для оси X при неравномерном шаге, `*.tx` для характеристик.

Для RecorderLnx это не означает немедленную реализацию совместимости, но формат записи надо проектировать так, чтобы не закрыть путь к MERA/USML-конвертерам.

## Плагины и обработка

Стандартные плагины Recorder из руководства:

- амплитудный спектр/БПФ в темпе эксперимента;
- компенсация температуры холодного спая;
- расчетные параметры;
- отчет о конфигурации.

Есть пользовательские подпрограммы/формулы:

- программа обработки состоит из списка подпрограмм;
- подпрограммы импортируются/экспортируются и могут редактироваться как текст с сохранением служебных комментариев;
- у подпрограммы есть свойства, включая название и период расчета;
- редактор формул содержит панель редактирования, список переменных каналов, операторы и функции.

Следствие для архитектуры: plugin/event API должен поддержать периодические вычисления по данным каналов, доступ к переменным каналов, регистрацию собственных формуляров/графиков, импорт/экспорт настроек обработки.

## Метрология и целостность

В Windows Recorder отдельно выделена метрологически значимая часть ПО: преобразование кодов АЦП в электрические или физические величины, контрольная сумма CRC32 доступна в окне информации.

Для RecorderLnx это будущий архитектурный слой:

- явно отделить метрологически значимые преобразования от UI и плагинов;
- хранить версии/контрольные суммы таких модулей;
- не смешивать экспериментальные плагины с метрологическим ядром.

## Влияние на ближайший план

До реализации `TRecorderStateMachine` нужно согласовать:

- имена состояний: `Stop`, `Preview`, `Record` вместо `Stop`, `View`, `Record`;
- нужен ли отдельный `Armed/WaitingForTrigger`;
- какие события state machine поддерживает в первой версии: `StartPreview`, `StartRecord`, `Stop`, `StartConditionMet`, `Timeout`, `Fault`;
- где хранить настройки условий старта/остановки: отдельный класс `TRecorderRunControlSettings` или пока record в unit state machine.
