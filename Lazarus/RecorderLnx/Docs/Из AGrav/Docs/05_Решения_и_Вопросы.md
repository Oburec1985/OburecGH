## Расположение исходников и тестов

Маркер решения: `RLNX_PATHS_2026_05_22`.

- Основной проект RecorderLnx: `D:\works\OburecGH\Lazarus\RecorderLnx\`.
- Общий код для разных Lazarus-проектов: `D:\works\OburecGH\Lazarus\SharedUtils\`.

Правило использования `SharedUtils`: перед созданием общего класса проверять существующие units в `SharedUtils`; если класс становится общим для нескольких проектов, переносить его туда только после отделения от доменной логики RecorderLnx.

Правило тестов: каждый новый класс RecorderLnx получает пример в `Tests\RecorderTests`; пример должен быть достаточно понятным, чтобы служить документацией использования.

- Пишем на Lazarus/FPC как нативное приложение.
- Целевые платформы первой версии: Windows и Linux.
- UI строится на LCL.
- Ядро не зависит от Windows COM.
- Зависимости допустимы только при наличии исходного кода и понятной лицензии.
- Расширяемость проектируется заранее, но первая реализация может начинаться со статически подключённых модулей, если публичный интерфейс не мешает будущим `.dll/.so`.
- Каждый новый класс (или несколько классов взаимосвязанных) сопровождается примером использования/тестом. Иногда стоит нарастить существующий тест, а не писать новый. Например, если сделали пример для добавления на наш компонент отрисовки графиков линии и сделали тест-пример, то потом при добавлении управления толщиной линии не стоит писать новый пример.
- Все классы и интерфейсы заносятся в проектный реестр, чтобы не дублировать функционал.


## Кроссплатформенность Windows/Linux

Маркер решения: `RLNX_CROSS_PLATFORM_2026_05_25`.

RecorderLnx должен собираться и работать и под Windows, и под Linux. Linux остается важной целевой средой эксплуатации, но архитектурные решения, структура проекта, тесты и зависимости не должны без необходимости привязывать ядро к одной ОС.

Платформенно-зависимый код допускается только за тонкими адаптерами с условной компиляцией `{$IFDEF WINDOWS}` / `{$IFDEF UNIX}` и отдельной записью в карте классов/модулей.

## Минимальная модель state machine

Маркер решения: `RLNX_STATE_MACHINE_2026_05_25`.

Первая реализация `TRecorderStateMachine` фиксирует рабочие режимы из руководства Recorder без каналов и файлов записи: `Stop`, `PreviewArmed`, `Preview`, `RecordArmed`, `Record`. Суффикс `Armed` означает, что пользователь запросил Preview/Record, но фактический старт ожидает условия: уровень сигнала, внешний цифровой/TTL trigger или время. Ручной старт сразу переводит в `Preview` или `Record`.

Прямой `Stop -> Record` разрешен, потому что в Recorder запись запускается F2; если задано неручное условие старта, переход идет в `RecordArmed`.

## Фабрики компонентов формуляров

Маркер решения: `RLNX_FACTORY_MANAGER_2026_05_25`.

Страница формуляра удаляет компоненты через их фабрику, чтобы не появлялись висячие ссылки.

## Расширяемость RecorderLnx

Маркер решения: `RLNX_EXTENSION_ARCHITECTURE_2026_05_26`.

RecorderLnx должен позволять модулям наращивать функционал так же широко, как Recorder: toolbar-ы, формуляры, компоненты мнемосхем, обработка данных, источники данных, сериализация, конфигурирование, запись и реакции на события состояния.

Принято:

- сохраняем знакомую механику `Register/Unregister`, `Factory`, `Notify`, `Start/Stop`, `ReadSettings/WriteSettings`;
- не переносим COM/ATL/MFC, `HWND`, `BSTR`, `VARIANT`, `IPicture` в core;
- первая реализация может быть статическими Lazarus/FPC-модулями;
- внешние `.dll/.so` проектируем через C ABI без FPC managed types на границе;
- плагины получают `TRecorderHostApi` как набор узких сервисов, а не неограниченный доступ ко всему core.

Планируемые сервисы расширяемости:

- `TRecorderExtensionManager`;
- `TRecorderEventBus`;
- `TRecorderActionRegistry`;
- `TRecorderToolbarRegistry`;
- `TRecorderVisualFactoryRegistry`;
- `TRecorderFormRegistry`;
- `TRecorderConfigRegistry`;
- `TRecorderProcessingRegistry`;
- `TRecorderDataSourceRegistry`;
- `TRecorderStorageRegistry`.

Имена классов и методов по возможности делаем аналогичными Recorder, если это не портит Pascal-архитектуру и кроссплатформенность.

Первая реализация core-сервисов создана в `D:\works\OburecGH\Lazarus\RecorderLnx\Core\uRecorderCoreServices.pas`:

- `TRecorderEventBus` — синхронная типизированная шина событий.
- `TRecorderActionRegistry` — реестр команд для будущих toolbar/menu/hotkeys.
- `IRecorderExtension` и `TRecorderExtensionManager` — lifecycle статических расширений первой версии.

Ограничение решения: шина пока синхронная. События из потоков источников данных нельзя напрямую использовать для изменения LCL; нужен будущий UI dispatcher.

## Первый контракт источников данных

Маркер решения: `RLNX_MOCK_DATA_SOURCE_2026_05_26`.

Первая модель источника данных реализована в `D:\works\OburecGH\Lazarus\RecorderLnx\Core\uRecorderDataSources.pas`.

Принято:

- `IRecorderDataSource` задает минимальный lifecycle: `ConfigureTags`, `Start`, `Stop`, `Tick`.
- `TRecorderDataSourceBase` хранит `SourceId`, имя, `UpdateTimeMs`, состояние и ссылку на `TRecorderTagRegistry`.
- `TMockSineDataSource` является первым отладочным источником: создает тег и публикует синус через `TRecorderTagRegistry.PublishValue`.
- `Tick` может вызываться вручную тестом или через `TRecorderDataSourceThread`.
- `TRecorderDataSourceThread` — runner одного источника. Он не является менеджером всех источников; его задача только безопасно вызвать `Start`, циклически выполнять `Tick` с `UpdateTimeMs`, реагировать на `Terminate` и вызвать `Stop` при выходе.
- `TRecorderDataSourceManager` — manager группы источников. Он хранит interface-ссылки на источники, настраивает теги через `ConfigureTagsAll`, запускает отдельный `TRecorderDataSourceThread` на каждый источник и останавливает группу в два этапа: `Terminate` всем, затем `WaitFor` всем.

Ограничение решения: события данных из `TRecorderDataSourceThread` публикуются в worker-thread. UI не должен менять LCL-компоненты из этих событий напрямую; нужен будущий UI dispatcher/queue.

## Настройка спектральных алгоритмов и частотные полосы

Маркер решения: `RLNX_SPECTRUM_UI_BANDS_2026_06_06`.

Диалог настройки RecorderLnx на вкладке `Каналы` поддерживает группу алгоритмов `Спектр`: при добавлении/drag-and-drop нескольких выбранных каналов в корень дерева создается один родительский узел спектра, а выбранные каналы становятся дочерними привязками этого узла. Если выбран уже существующий узел спектра, новые каналы добавляются в него без дублей.

Для спектра в UI отображаются параметры, которые относятся к расчету, а не к визуализации: размер FFT, частота опроса, расчетный размер порции в секундах, количество блоков усреднения, режим перекрытия (`без перекрытия`, `1/2`, `1/4`), окно, zero padding, коррекция АЧХ и режим интегрирования (`без`, однократное, двухкратное). Нормировка больше не является параметром пользовательского диалога: тип оценки/представления должен выбирать слой визуализации, а ядро оставляет стабильную RMS-оценку.

Коррекция АЧХ в первом UI-варианте фиксируется как включение профиля коэффициентов. Будущая core-реализация должна определить функцию, которая домножает каждый спектральный сэмпл на калибровочный коэффициент.

Частотные полосы выделены в отдельную core-абстракцию `TRecorderFrequencyBand`: полосы могут быть абсолютными в Гц или формульными. Формульная полоса повторяет полезную идею `plgControlCyclogram/uBaseAlgBands.pas`: базовая частота считается как сумма `k * значение_тахо_или_частотного_тега`, затем границы применяются как относительные множители. Диалог `uRecorderFrequencyBandsDialog.pas` является только редактором модели; спектры, АЧХ и графики должны линковаться к тем же объектам полос без зависимости от формы.

- UI получает `TRecorderTimeSnapshot.DisplayText` и только отображает его в транспаранте;
- `DisplayUpdateMs` фиксирует период обновления отображения и будет использоваться будущим display-thread;
- `TStringList` для последних значений тегов в `UI\uMainForm.pas` не должен быть `Sorted`, потому что `Values[...] :=` в FPC/Lazarus падает на sorted-list. Это исправляет ошибку перехода в просмотр.
