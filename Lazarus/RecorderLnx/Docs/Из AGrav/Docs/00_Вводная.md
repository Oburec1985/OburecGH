# RecorderLnx: вводная и границы проекта

## Назначение

RecorderLnx — новая нативная система сбора, отображения, записи и обработки измерительных данных под Windows и Linux. Пишем на Lazarus/FPC, сохраняя проверенные архитектурные идеи Windows Recorder, но не копируя механически COM-зависимости и Windows API.

Система должна работать с разными измерительными системами. Поэтому в ядре проекта сразу вводится понятие `источник данных`: узел дерева конфигурации, который знает свои настройки, период обновления, состояние связи и способ публикации измеренных значений в общую модель данных.

## Базовая модель работы

Приложение находится в одном из основных режимов:

- `Stop` — безопасное конфигурирование. Можно менять дерево устройств/источников, формуляры, плагины, параметры записи.
- `View` — сбор и отображение данных. Потоки источников работают, UI получает снимки данных и перерисовывает формуляры.
- `Record` — сбор, отображение и запись. Все правила `View` сохраняются, дополнительно активны писатели данных и контроль записи.

Windows Recorder также имел состояния воспроизведения, паузы, калибровки, ошибок, потерь пакетов, необходимости reset/config. В RecorderLnx эти состояния не выбрасываются, но вводятся постепенно через типизированную машину состояний, а не через россыпь флагов без владельца.

## Главные ограничения

- Использовать при сборке только библиотеки, к которым есть исходный код и понятная лицензия.
- Не создавать зависимость от неподдерживаемых закрытых компонентов.
- Любая новая библиотека или крупный класс фиксируется в [Карте интерфейсов](./03_Карта_Интерфейсов.md) или отдельном реестре проекта.
- Каждый класс или крупный кусок функционала сопровождается примером использования, который одновременно является тестом или проверочным сценарием.
- Работа идёт маленькими проверяемыми шагами: проектирование → код → пример/тест → документация.

## Что изучено из Windows Recorder

Исходные интерфейсы находятся в `D:\works\OburecGH\recorder\SharedRUnits\interfaces`.

Ключевые идеи, которые переносим концептуально:

- `IRecorder` как центральный фасад приложения: состояние, теги, группы, устройства, формуляры, плагины, свойства, окружение, уведомления.
- `ITag` и `ITagsGroup` как модель измерительных каналов и группировки каналов.
- `IDevice` / `IModule` как слой аппаратуры и модулей.
- `IRecorderPlugin` и `PN_*` уведомления как событийная модель расширений.
- `IVForm` / `ICustomFormFactory` как механизм внешних формуляров отображения.
- `ISignal`, шкалы и трансформеры как слой данных, тарировки и пересчёта.
- `IAlarm*` как слой уставок, событий и триггеров записи.

## Что не переносим напрямую

- Windows COM как основной механизм расширяемости.
- `HWND`, `LPCSTR`, `BSTR`, `OleVariant` как доминирующие типы внутри ядра.
- Windows-only units (`Windows`, `ActiveX`, `ComObj`) в кроссплатформенном ядре.
- Неявную работу с указателями там, где можно сделать типизированные FPC-интерфейсы или классы.
