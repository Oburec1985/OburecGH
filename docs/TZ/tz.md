При запуске ЦАП потом останове, потом следующем запуске происходит ошибка
raise Exception.Create('Error writing to waveOut device: ' + IntToStr(ResultCode)); (uSoundCardDAC)
Давай исправим.

**Решение:**
1. Добавлен сброс `FQueuedBuffers := 0` в методе `Start` перед заполнением очереди
2. Добавлен сброс `FQueuedBuffers := 0` в методе `Stop` для остановки постановки новых буферов
3. В `QueueBuffer` изменена логика - при отсутствии свободных буферов теперь `exit` вместо исключения
4. Флаги цикла (`WHDR_BEGINLOOP`, `WHDR_ENDLOOP`) устанавливаются **после** `waveOutPrepareHeader`
5. В `Open` инициализируется `FCurrentLoopCount := 1` по умолчанию
6. Callback проверяет `FDeviceHandle = 0` для защиты от гонок при закрытии
7. Добавлен метод `TBlockQueue.GetBlockIndex` для получения индекса блока по указателю
8. Добавлен метод `TDacDevice.GetBlockIndex` для доступа из наследников

**Причина проблемы:**
- Счетчик буферов `FQueuedBuffers` не сбрасывался перед каждым запуском
- Флаги цикла устанавливались до `waveOutPrepareHeader`, что вызывало ошибку 33 (MMSYSERR_INVALPARAM)
- При повторном запуске из состояния `stOpened` использовались старые буферы без очистки
- Очередь блоков не очищалась между запусками

**Итог:**
ЦАП теперь корректно работает в режиме: запуск ? останов ? запуск (многократно)

Задание 3. Статус: **Выполнено**
**Задача: Создание модуля управления сценариями ЦАП (uDACProgram.pas) на Delphi.
Контекст:У меня есть система вывода звука (uDacDevice, uSoundCardDac), работающая через waveOut.

**Архитектурные требования:
3.1. Базовый класс TDacProgram:
- Содержит ссылку на fDevice: TDacDevice.
- Хранит fCurrentPhase: Double (для обеспечения преемственности фазы между буферами).
- Имеет виртуальный метод ProcessBuffer(P: Pointer; Size: Integer), который вызывается внутри обработчика OnBufferEnd.
- Содержит метод для расчета «выравнивания» буфера под частоту (GetAlignedSize(Freq: Double; MinSamples: Integer): Integer.

3.2. Наследник TSimpleSinusProgram (Тупое управление):
- Реализует простую выдачу синуса по текущим параметрам Frequency и Amplitude.
- Важно: Сохраняет текущую фазу, чтобы не было разрывов внутри одного потока, но НЕ меняет размер буфера устройства (использует текущий fDevice.BufferSize).
Это «базовое поведение» без подстройки под кратность периодов.

3.3. Наследник TAccurateSinusProgram (Умное управление):
- При изменении частоты автоматически вычисляет новый размер буфера, кратный периоду (используя MinSamplesTarget).
- Вызывает fDevice.ReallocateBuffers(NewSize), если это необходимо.
- Гарантирует, что в буфере всегда целое число периодов.

**Технические детали для реализации:
- Модуль должен использовать типы из uCommonTypes (например, PSmallIntArray).
- Реализуй механизм подписки программы на события устройства: программа должна сама назначать свой метод на fDevice.OnBufferEnd.
- Предусмотри защиту: если частота $0$ или слишком мала, размер буфера не должен превышать разумные пределы, или переходи на логику с тупым поведеним когда надо склеивать несколько буферов.
- Напиши пример кода для Unit1, как инициализировать программу и переключаться между TSimpleSinusProgram и TAccurateSinusProgram.
